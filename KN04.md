### Was ist das Hauptmerkmal einer Time-Series Database?

Eine Time-Series Database wie Prometheus speichert jeden Datenpunkt zusammen mit einem Zeitstempel – so kann man den Verlauf von Werten über Zeit abfragen und visualisieren. Im Gegensatz zu einer relationalen Datenbank ist sie optimiert fuer das Schreiben und Lesen von zeitlich geordneten Metriken, nicht fuer komplexe JOIN-Abfragen oder Transaktionen.

### Warum sind Terminal-Logs in einer Cloud-Umgebung eine schlechte Idee?

Bei hunderten von Servern muesste man sich manuell in jeden einzelnen Server einloggen um die Logs zu lesen – das ist nicht skalierbar. Ausserdem gehen Logs verloren wenn eine Instanz abstuerzt oder neu gestartet wird, genau dann wenn man sie am meisten braeuchte. Ein zentrales Monitoring-System wie Prometheus sammelt alle Daten automatisch und macht sie ueber ein einziges Dashboard zugaenglich.

## Phase 2: Die Instrumentierung (Code)

### Angepasster Python-Code

```python
import time
import random
from prometheus_client import start_http_server, Counter

upload_metric = Counter('file_uploads_total', 'Anzahl der Uploads', ['dateityp', 'status'])

def process_upload():
    file_types = ['image/png', 'image/jpeg', 'video/mp4']
    file_type = random.choice(file_types)

    if random.random() > 0.1:
        status = 'success'
        print(f"[UPLOAD] {file_type} erfolgreich gespeichert.")
    else:
        status = 'error_file_too_large'
        print(f"[UPLOAD] FEHLER: {file_type} ist zu gross!")

    upload_metric.labels(dateityp=file_type, status=status).inc()

if __name__ == '__main__':
    print("Starte Datei-Upload Simulation...")
    start_http_server(8000)
    print("Metrik-Server laeuft auf Port 8000")

    while True:
        process_upload()
        time.sleep(random.uniform(0.2, 1.5))
```

### Erklärung Aggregation

Bei einer kurzen Zeitspanne (5 Minuten) sieht man jeden einzelnen Datenpunkt als scharfe Kurve mit vielen Details. Bei einer langen Zeitspanne (1 Stunde) werden die Datenpunkte zusammengefasst (aggregiert), was die Linie glatter und weniger detailliert macht.

## Phase 5: Architektur-Reflexion

### Warum localhost als Target-URL?

Prometheus und Grafana laufen auf demselben Server wie das Python-Script. Aus Sicht des Servers ist sich selbst immer `localhost` – egal welche Public-IP von aussen verwendet wird. Die Public-IP ist nur fuer den Zugriff von aussen (Browser/SSH) relevant, intern kommunizieren die Dienste ueber localhost.

### Vorteil von Labels gegenüber separaten Tabellen?

Mit Labels kann man eine einzige Metrik (`file_uploads_total`) nach beliebigen Dimensionen filtern und gruppieren – z.B. nach `dateityp` oder `status`. In SQL bräuchte man dafür separate Tabellen oder komplexe JOINs. Mit Labels schreibt man einfach `file_uploads_total{status="error"}` und bekommt sofort nur die Fehler angezeigt.

### Woher weiss Prometheus wann ein Ereignis stattgefunden hat?

Im Pull-Modell scrapt Prometheus in regelmässigen Abständen (z.B. alle 15 Sekunden) den `/metrics` Endpoint des Python-Scripts. Dabei fügt Prometheus selbst den aktuellen Zeitstempel hinzu – nicht die Applikation. So entsteht eine Zeitreihe: Prometheus weiss zu welchem Zeitpunkt er die Daten abgeholt hat und speichert diesen Wert zusammen mit dem Metrik-Wert in seiner Time-Series Database.
