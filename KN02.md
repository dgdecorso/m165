## Phase 2: Das relationale Erbe

### Code – Aggregation Pipeline

```js
db.getCollection('posts').aggregate([
  {
    $lookup: {
      from: "users",
      localField: "user_id",
      foreignField: "_id",
      as: "author"
    }
  },
  {
    $unwind: "$author"
  },
  {
    $lookup: {
      from: "comments",
      localField: "_id",
      foreignField: "post_id",
      as: "comments"
    }
  }
])
```

### Erklärung

Die Pipeline geht von der `posts`-Collection aus. Mit dem ersten `$lookup` wird der passende User aus `users` anhand der `user_id` dazugeholt und als `author` gespeichert. Da `$lookup` ein Array zurückgibt, wird mit `$unwind` daraus ein einzelnes Objekt gemacht. Der zweite `$lookup` holt dann alle Kommentare aus `comments` die zur `post_id` gehoeren und haengt sie als Array ans Dokument.

## Phase 3: Die Performance-Analyse

### Metriken

| Metrik | Wert |
|---|---|
| `totalDocsExamined` | 49 (5 posts + 4 users + 40 comments) |
| `nReturned` | 5 |
| `executionTimeMillis` | 0 |

### Beweis

![explain executionStats Phase 3](SCREENSHOT_PHASE3_EXPLAIN.png)

### Fazit

Das Problem ist vor allem beim zweiten `$lookup` auf die `comments`-Collection sichtbar. Da kein Index auf `post_id` existiert, macht MongoDB einen Collection Scan – also schaut er sich bei jedem der 5 Posts alle 8 Kommentare an, was zu 40 examinierten Dokumenten fuehrt. Bei unserem kleinen Testdatensatz ist das noch kein Problem, aber wenn man z.B. 100'000 Posts und Millionen Kommentare haette, wuerde die DB fuer jeden einzelnen Post die gesamte Comments-Collection durchsuchen. Das ist ein klassischer Flaschenhals: `totalDocsExamined` waechst linear mit den Daten, aber `nReturned` bleibt klein.

## Phase 4: Der Paradigmenwechsel (Design)

### Neue JSON-Struktur (Embedded Document)

```json
{
  "_id": "post_1",
  "content": "Heute war ein super Tag im Park! ☀️",
  "imageUrl": "https://example.com/img1.jpg",
  "likes": 42,
  "createdAt": "2025-05-10T08:30:00Z",
  "author": {
    "user_id": "user_1",
    "username": "lena_m",
    "displayName": "Lena Müller",
    "followerCount": 340
  },
  "comments": [
    {
      "comment_id": "comment_1",
      "user_id": "user_2",
      "username": "marco_b",
      "displayName": "Marco Bianchi",
      "text": "Klingt wunderbar! Wo war das?",
      "createdAt": "2025-05-10T09:00:00Z"
    },
    {
      "comment_id": "comment_2",
      "user_id": "user_3",
      "username": "sara_k",
      "displayName": "Sara Koch",
      "text": "Ich war auch dort, so schoen! 😍",
      "createdAt": "2025-05-10T09:45:00Z"
    }
  ]
}
```

### Begründung

Ich hab den `author` und die `comments` direkt ins Post-Dokument eingebettet. Den Author hab ich eingebettet weil er immer zusammen mit dem Post geladen wird – es macht keinen Sinn ihn separat zu speichern wenn man ihn sowieso jedes Mal braucht. Bei den Kommentaren ist es aehnlich: ein Post hat normalerweise nicht tausende Kommentare, also ist das eine 1-to-Few Beziehung und Embedding passt da gut. Im Social Media Feed wird viel mehr gelesen als geschrieben, das heisst das Read-to-Write Ratio ist hoch. Deshalb lohnt es sich die Daten so zu speichern dass man alles mit einem einzigen `find()` bekommt, auch wenn dafuer z.B. der Username doppelt gespeichert wird. Wenn sich mal ein Username aendert muesste man das manuell aktualisieren, aber das passiert selten.

## Phase 5: Der Beweis

### Code – find() Abfrage

```js
db.getCollection('feed').find({ "_id": "post_1" })
```

### Metriken (Vergleich)

| Metrik | Phase 3 (alt) | Phase 5 (neu) |
|---|---|---|
| `totalDocsExamined` | 49 | 3 |
| `nReturned` | 5 | 3 |
| `executionTimeMillis` | 0 | 0 |

### Beweis

![explain executionStats Phase 5](SCREENSHOT_PHASE5_EXPLAIN.png)

### Vergleich

Im alten Design musste MongoDB insgesamt 49 Dokumente ueber 3 verschiedene Collections durchsuchen um 5 Posts mit Author und Kommentaren zusammenzustellen. Mit dem neuen eingebetteten Design reichen 3 Dokumente in einer einzigen Collection – kein `$lookup` mehr noetig. Das Re-Design hat sich klar gelohnt, weil bei steigenden Datenmengen der Unterschied noch viel groesser werden wuerde.
