## Phase 2: Kanten-Attribute (Relationship Properties)
 
### Vollständiger CREATE Codeblock
 
```cypher
CREATE 
  (acc1:Account {id: '1001', owner: 'Alice'}),
  (acc2:Account {id: '1002', owner: 'Bob'}),
  (acc3:Account {id: '1003', owner: 'Charlie'}),
  (acc4:Account {id: '1004', owner: 'Dave'}),
  (acc1)-[:TRANSFERRED {amount: 5000}]->(acc2),
  (acc2)-[:TRANSFERRED {amount: 3000}]->(acc3),
  (acc3)-[:TRANSFERRED {amount: 7000}]->(acc1),
  (acc3)-[:TRANSFERRED {amount: 2000}]->(acc4)
```

## Phase 3: Der "Broken AI" Ansatz (Debugging)
 
### Logischer Fehler der ursprünglichen Query
 
Die urspruengliche Query hat den Pfeil in die falsche Richtung – sie sucht nach Accounts die an Charlie ueberwiesen haben (`<-`), statt nach Accounts an die Charlie ueberwiesen hat (`->`). Ausserdem wird `recipient.amount` abgefragt, aber `amount` ist ein Attribut auf der Kante (Relationship), nicht auf dem Knoten.
 
### Fehlerhafte Query (Original)
 
```cypher
MATCH (charlie:Account {owner: 'Charlie'})<-[:TRANSFERRED]-(recipient:Account)
RETURN recipient.owner, recipient.amount
```
 
### Reparierte Query
 
```cypher
MATCH (charlie:Account {owner: 'Charlie'})-[t:TRANSFERRED]->(recipient:Account)
RETURN recipient.owner, t.amount
```

## Phase 4: Architektur-Reflexion
 
### SQL-Modell für die gleichen Kanten-Attribute
 
In SQL bräuchte man drei Tabellen:
 
- **Account** (id PK, owner)
- **Transfer** (id PK, from_account_id FK → Account.id, to_account_id FK → Account.id, amount)
Die `Transfer`-Tabelle ist die Mapping-Tabelle die die Beziehung zwischen zwei Accounts speichert, inklusive dem `amount` als zusaetzliches Attribut. In Neo4j ist das einfach eine Kante mit einem Property – kein zusaetzliche Tabelle noetig.
 
### Warum ist Neo4j bei tiefen Beziehungen schneller als SQL?
 
In SQL muss fuer jeden "Hop" im Netzwerk ein JOIN ausgefuehrt werden – bei 4 Hops bedeutet das 4 verschachtelte JOINs, die immer groessere Zwischentabellen erzeugen und exponentiell langsamer werden. Neo4j verwendet **Index-Free Adjacency**: jeder Knoten speichert direkt Zeiger auf seine Nachbarknoten. Um zum naechsten Knoten zu gelangen muss Neo4j nicht die gesamte Datenbank durchsuchen, sondern folgt einfach dem Zeiger. Das ist unabhaengig von der Datenbankgroesse immer gleich schnell.
