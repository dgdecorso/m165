## Phase 2: Die fehlerhafte KI-Tabelle

### CREATE TABLE & INSERT (fehlerhaft)

```cql
CREATE TABLE clicks (
  click_id uuid PRIMARY KEY,
  user_id text,
  product_id text,
  event_type text,
  clicked_at timestamp
);

INSERT INTO clicks (click_id, user_id, product_id, event_type, clicked_at)

INSERT INTO clicks (click_id, user_id, product_id, event_type, clicked_at)

INSERT INTO clicks (click_id, user_id, product_id, event_type, clicked_at)
```

### SELECT Befehl der das Problem ausloest

### Erklaerung: Warum wirft Cassandra diesen Fehler?

Cassandra ist ein verteiltes System, das Daten anhand des Partition Keys auf verschiedene Nodes verteilt (Sharding). In der fehlerhaften Tabelle ist `click_id` der Partition Key – das heisst, Cassandra weiss nicht auf welchem Node die Daten fuer `user_id = 'user_42'` liegen. Um die Abfrage auszufuehren muesste Cassandra alle Nodes gleichzeitig durchsuchen (Full Cluster Scan), was bei Millionen von Zeilen extrem langsam waere. Deshalb verweigert Cassandra die Abfrage mit dem Fehler `Cannot execute this query as it might involve data filtering`. In MySQL gibt es dieses Problem nicht, weil MySQL eine einzelne Maschine ist und einfach einen Full Table Scan machen kann.

## Phase 3: Query-First Design (Modellierung reparieren)

### DROP TABLE

```cql
DROP TABLE clicks;
```

### Korrigierter CREATE TABLE

Der Partition Key muss `user_id` sein, weil wir immer nach einem bestimmten User abfragen. Die Clustering Column ist `clicked_at DESC`, damit die neuesten Clicks zuerst kommen.

```cql
CREATE TABLE clicks_by_user (
  user_id text,
  clicked_at timestamp,
  click_id uuid,
  product_id text,
  event_type text,
  PRIMARY KEY (user_id, clicked_at)
) WITH CLUSTERING ORDER BY (clicked_at DESC);
```

### INSERT Daten neu laden

```cql
INSERT INTO clicks_by_user (user_id, clicked_at, click_id, product_id, event_type)
VALUES ('user_42', toTimestamp(now()), uuid(), 'prod_99', 'view');

INSERT INTO clicks_by_user (user_id, clicked_at, click_id, product_id, event_type)
VALUES ('user_42', toTimestamp(now()), uuid(), 'prod_55', 'add_to_cart');

INSERT INTO clicks_by_user (user_id, clicked_at, click_id, product_id, event_type)
VALUES ('user_77', toTimestamp(now()), uuid(), 'prod_99', 'purchase');
```

### SELECT funktioniert nun

```cql
SELECT * FROM clicks_by_user WHERE user_id = 'user_42' ORDER BY clicked_at DESC;
```

## Phase 4: Architektur-Reflexion

### Warum ist ALLOW FILTERING ein Anti-Pattern?

`ALLOW FILTERING` sagt Cassandra einfach, dass es den Fehler ignorieren und trotzdem alle Nodes durchsuchen soll. Bei einer kleinen Testdatenbank mit 3 Zeilen ist das kein Problem. Aber bei Netflix oder Spotify mit Milliarden von Clickstream-Eintraegen wuerde das bedeuten, dass jede Abfrage Hunderte von Servern gleichzeitig abfragen und riesige Datenmengen ueber das Netzwerk schicken muss – das kann die gesamte Cluster-Performance zerstoeren und zu Timeouts fuehren. Deshalb ist es in der Produktion absolut verboten.

### Unterschied Partition Key vs. Clustering Column

Der **Partition Key** bestimmt, auf welchem Node die Daten gespeichert werden. Alle Zeilen mit dem gleichen Partition Key landen auf dem gleichen Node – das ist der Shard. Cassandra kann deshalb direkt zum richtigen Node gehen ohne andere zu fragen.

Die **Clustering Column** bestimmt, wie die Daten innerhalb einer Partition sortiert werden. Sie wird auf dem gleichen Node gespeichert und erlaubt effiziente Range-Queries und Sortierungen innerhalb einer Partition. Im Clickstream-Beispiel: `user_id` ist der Partition Key (alle Clicks eines Users auf einem Node), `clicked_at` ist die Clustering Column (innerhalb dieses Users nach Zeit sortiert).
