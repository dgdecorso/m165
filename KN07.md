## Phase 1: Setup & Das Embedding-Wunder

### setup_db.py

```python
import chromadb

client = chromadb.PersistentClient(path="./chroma_data")
collection = client.get_or_create_collection(name="movies")

documents = [
    "Ein Milliardär baut einen Anzug aus Metall und rettet die Welt.",
    "Ein junger Zauberer entdeckt seine magischen Fähigkeiten an einer Internatsschule.",
    "Eine Gruppe von Dieben dringt in die Träume von Menschen ein, um Geheimnisse zu stehlen.",
    "Ein einsamer Roboter räumt die verlassene Erde auf und verliebt sich."
]
ids = ["movie_1", "movie_2", "movie_3", "movie_4"]

collection.add(documents=documents, ids=ids)
print("Film-Plots wurden erfolgreich vektorisiert und gespeichert!")
```

### Was passiert im Hintergrund beim Speichern?

Beim `collection.add()` Aufruf schickt ChromaDB jeden Text automatisch durch ein Embedding-Modell (standardmässig `all-MiniLM-L6-v2`), das den Text in einen Vektor mit mehreren hundert Zahlen umwandelt. Dieser Vektor repräsentiert die Bedeutung des Texts im Vektorraum – Texte mit ähnlicher Bedeutung landen nahe beieinander. Dieser Vektor wird zusammen mit dem Originaltext gespeichert, nicht nur der reine Text.

## Phase 2: Die fehlerhafte KI-Abfrage (Lexikalisch vs. Semantisch)

### Fehlerhafter Code (search_db.py)

```python
import chromadb

client = chromadb.PersistentClient(path="./chroma_data")
collection = client.get_collection(name="movies")

suchanfrage = "Der Film, in dem jemand in den Schlaf anderer Leute einbricht"

print(f"Suche nach Film: '{suchanfrage}'...")

# FEHLERHAFTER KI-CODE: Sucht nach exaktem Metadaten-Match (wie SQL)
resultate = collection.get(
    where={"text": suchanfrage}
)

print("Gefundene Filme:", resultate["documents"])
```

### Warum lieferte get(where=...) keine Resultate?

`get(where=...)` ist ein reiner Metadaten-Filter, der nach exakter Übereinstimmung sucht – wie `WHERE` in SQL. Beim Speichern in Phase 1 wurde gar kein `text` Metadaten-Feld mitgegeben, und selbst wenn, müsste der Suchstring buchstabengetreu mit einem gespeicherten Wert übereinstimmen. Da die Suchanfrage "Der Film, in dem jemand in den Schlaf anderer Leute einbricht" nirgends wortwörtlich vorkommt, findet `get()` nichts – obwohl der Plot von movie_3 inhaltlich genau passt.

### Korrigierter Code (search_db.py)

```python
import chromadb

client = chromadb.PersistentClient(path="./chroma_data")
collection = client.get_collection(name="movies")

suchanfrage = "Der Film, in dem jemand in den Schlaf anderer Leute einbricht"

print(f"Suche nach Film: '{suchanfrage}'...")

resultate = collection.query(
    query_texts=[suchanfrage],
    n_results=2
)

print("Gefundene Filme:", resultate["documents"])
```

## Phase 3: Architektur-Reflexion

### Warum erkennt eine Vektordatenbank "Hund" ≈ "Welpe", SQL aber nicht?

Das Embedding-Modell wurde mit riesigen Textmengen trainiert, in denen "Hund" und "Welpe" oft im selben Kontext auftauchen (Tier, Fell, Haustier, bellen...). Dadurch landen ihre Vektoren im hochdimensionalen Vektorraum nahe beieinander – die Distanz zwischen ihnen ist klein. ChromaDB misst bei einer Suche genau diese Distanz (z.B. Cosine-Similarity) und gibt die nächstgelegenen Treffer zurück. SQL mit `LIKE '%Hund%'` macht dagegen reinen Zeichenkettenvergleich: "Welpe" enthält die Buchstaben "Hund" nicht, also gibt es keinen Treffer, egal wie ähnlich die Bedeutung ist.

### Warum oft semantische Suche (query) mit Metadaten-Filter (where) kombinieren?

Reine semantische Suche findet zwar inhaltlich passende Treffer, kann aber keine harten Einschränkungen garantieren. Bei der Film-Datenbank will man z.B. nur Treffer aus einem bestimmten Genre oder ab einem bestimmten Jahr. Kombiniert man `query` mit `where`, bekommt man das Beste aus beiden Welten:

```python
collection.query(
    query_texts=["Roboter rettet die Zukunft"],
    where={"genre": "Sci-Fi", "jahr": {"$gte": 2010}},
    n_results=5
)
```

So liefert ChromaDB die semantisch ähnlichsten Filme, aber gefiltert auf exakte Kriterien, die rein über Bedeutung nicht zuverlässig erfassbar sind.
